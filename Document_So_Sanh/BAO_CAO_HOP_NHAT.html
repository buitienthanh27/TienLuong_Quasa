1. Executive Summary

Quasa và Mekong là hai công ty cao su thuộc cùng chủ sở hữu (Geruco/công ty mẹ Việt Nam), nhưng vận hành trên hai quốc gia khác biệt (Lào & Campuchia), với hai triết lý tính lương về bề ngoài rất khác: Quasa trả thẳng theo tiền (đơn giá × sản lượng), Mekong quy mọi thứ về công rồi nhân ngược lại ra tiền. Tuy nhiên, khi phân tích sâu ta thấy mô hình toán học bên dưới chỉ là một — đều là "quy đổi đầu vào về một đơn vị chuẩn, rồi nhân với giá chuẩn". Khác nhau chỉ là: giá chuẩn là 1 kg mủ × hệ số hạng (Quasa) hay 1 công = 31.947 Riel (Mekong).

Vì vậy, không cần xây hai phần mềm. Đề xuất của báo cáo này là xây một engine duy nhất (Unified Payroll Engine) với:

  • Một data model trung lập với khái niệm WorkUnit (có thể là kg, công, USD, kip, riel).
  • Một Rule Engine thực thi pipeline 5 bước (Normalize → Price → Allowance → Deduction → Net).
  • Hai bộ RuleSet config (JSON/YAML): quasa.yaml và mekong.yaml, chạy trên cùng 1 engine.
Giá trị kinh doanh
  • Giảm ~50% chi phí dev & maintain (1 codebase thay vì 2).
  • Khi thay đổi luật thuế/BHXH — chỉ cập nhật rule, không cần release.
  • Dễ mở rộng sang công ty cao su thứ 3, 4 trong tương lai.
Cam kết kỹ thuật
  • Zero-downtime khi đổi đơn giá/biểu thuế (rule versioning).
  • Full audit trail (ai sửa ô nào, khi nào).
  • Import file Excel cũ để migration → không mất dữ liệu lịch sử.

4. So sánh công thức

Mỗi nhóm công thức được trình bày ở 3 cột: công thức Quasa (trích nguyên văn từ Excel/SRS), công thức Mekong (trích từ 03_Cong_Thuc_Tinh_Luong.md), và pseudo-code unified dùng được cho cả 2 qua rule config.

4.1 Lương sản lượng (core)
QuasaMekongUnified DSL
 
M16 = H16*AE6 + K16*AE5
O16 = M16 × hệ_số_hạng × 600
// hệ_số: A=9.2, B=8.9, C=8.6, D=8.3 (Trạm 1)
// A=7.7, B=7.4, C=7.1, D=6.8 (Trạm 2)
 
mu_dong = mu_tuoi × 0.44   // DRC
cong_mu = ROUND(mu_dong × gia/31947, 2)
// gia: 455 | 505 | 555 KHR/kg
tien = tong_cong × 31947
 
// rule: production_to_money
qty_normalized = qty_raw × conversion_factor
money = qty_normalized × unit_price × fx_rate
// → Quasa: conversion_factor=DRC, fx=600
// → Mekong: conversion_factor=0.44, fx=1
//            (qua bước intermediate "cong")
4.2 Lương phụ cấp theo ngày công
QuasaMekongUnified DSL
 
Y16 = (P16×46.1 + Q16×76.9
     + R16×30.7
     + T16×46.1×2 + U16×76.9×2) × factor
     + S16×20000 + V16×100000
// factor = 667.96 (AD4), thay đổi mỗi tháng
 
cong_xang = ROUND(ngay × 1000/31947, 2)
cong_mi_dau = ROUND(ngay × 35000/31947, 2)
cong_chuyen_can = ROUND(ngay × 3000/31947, 2)
// tổng công → × 31947
 
// rule kind: ALLOWANCE_PER_DAY
for each allowance_rule:
  amount += days × rate_per_day × coef
// Quasa: coef=factor(monthly), rate=46.1/76.9...
// Mekong: coef=1/31947, rate=1000/35000/3000
// + finalize: tổng × 31947 (chỉ Mekong)

4.4 BHXH / Social security
QuasaMekongUnified DSL
 
// SRS: BHXH 8% + BHYT 2.5% (Lào)
bhxh = gross × 0.08
bhyt = gross × 0.025
// CN khai thác HIỆN KHÔNG bị trừ bảo hiểm
// (SRS 2.6: "không trừ thuế, không trừ bảo hiểm")
 
IF L15 ≤ 400000: bhxh = 400000 × 2%  // sàn
ELIF L15 ≤ 1200000: bhxh = L15 × 2%
ELSE: bhxh = 1200000 × 2%            // trần
// Chỉ ở Đợt 2, Đợt 1 = 0
 
// rule: STATUTORY_DEDUCTION
for each insurance_scheme:
  base = apply_cap(gross, floor, ceiling)
  amount = base × rate
  skip_if = rule.only_on_settlement_run
4.5 Thuế TNCN Phụ lục — GĐ2, chỉ cán bộ
⚠️ Section này KHÔNG áp dụng trong MVP Giai đoạn 1. Công nhân trực tiếp khai thác ở cả Quasa & Mekong đều không bị trừ TNCN. Nội dung dưới đây giữ lại để tham chiếu khi mở rộng sang lương cán bộ (GĐ2).
QuasaMekongUnified DSL
 
// Biểu Lào (LƯƠNG ĐỘI!J13, CB Lào)
if I13 > 65M: (I13-65M)×35% + 10.685M
elif I13 > 25M: (I13-25M)×25% + 2.685M
elif I13 > 20M: (I13-20M)×20% + 2.250M
elif I13 > 10M: (I13-10M)×15% + 750k
elif I13 >  5M: (I13- 5M)×10% + 250k
elif I13 > 1.3M: (I13-1.3M)×5%
else: 0
 
// Biểu Campuchia (Đợt 2, 2026)
taxable = L15 - K15(T13) - N15(BHXH)
if taxable ≤ 1.5M: 0
elif ≤ 2M: taxable×5% - 75k
elif ≤ 8.5M: taxable×10% - 175k
// >8.5M: THIẾU nhánh ⚠ (Cần bổ sung 15%/20%)
 
// rule: PROGRESSIVE_TAX
taxable = gross - exemptions
for bracket in brackets:
  if taxable ≤ bracket.cap:
    tax = taxable × bracket.rate - bracket.deduct
    break
4.6 Thực nhận (Net pay)
QuasaMekongUnified DSL
 
Z16 = O16 + G16 + Y16  // tổng lương
AA16 = Z16 - tax         // thực nhận
 
L15 = G15 + I15 + J15 + K15  // tổng cả tháng
P15 = M15(advance) + N15(BHXH) + O15(TNCN)
R15 = L15 - P15               // thực nhận Đợt 2
 
net = gross
    - sum(statutory_deductions)
    - sum(advances_in_period)
    - sum(other_deductions)

5. Điểm GIỐNG nhau

  • ✓ Cả 2 đều là payroll theo sản lượng mủ cao su, tính hàng tháng.
  • ✓ Input cơ bản giống nhau: chấm công ngày + sản lượng mủ ngày + phụ cấp.
  • ✓ Đều có bước quy đổi DRC từ mủ tươi/tạp → mủ khô/đông.
  • ✓ Đều phân loại công nhân theo chất lượng (hạng A/B/C/D ở Quasa vs cấp mủ 455/505/555 ở Mekong) — cùng ý tưởng: đơn giá khác theo chất lượng.
  • ✓ Đều có phụ cấp đặc biệt theo ngày công (khộp nặng, cây non, chuyên cần, PCCC…) — đều là rate_per_day × days.
  • ✓ Cán bộ (GĐ2) ở cả 2 công ty đều khấu trừ BHXH + TNCN lũy tiến theo luật sở tại. Công nhân trực tiếp khai thác (GĐ1): KHÔNG trừ TNCN ở cả Quasa & Mekong; BHXH tuỳ quy định từng công ty.
  • ✓ Đều có Bảng phân toán chia tổng quỹ theo nhóm chi phí (kế toán tập hợp chi phí).
  • ✓ Đều báo cáo lên công ty mẹ Việt Nam qua tỷ giá quy đổi (USD/VND).
  • ✓ Đều có hardcode factor thay đổi theo tháng (667,96 ở Quasa · 31.947 ở Mekong).
  • ✓ Đều có lỗi/rủi ro Excel: #REF!, external link, công thức thiếu nhánh lũy tiến cao.

Cơ chế đồng bộ đề xuất: (1) API pull theo kỳ (daily cron) cho production/timesheet; (2) Webhook push cho thay đổi hồ sơ; (3) Khoá kỳ lương (period_lock) — sau khi chạy payroll, dữ liệu tương ứng trong Sổ mủ không còn ảnh hưởng; chỉ tạo adjustment entry.

Nguyên tắc thiết kế

  • Separation of concerns: công thức → rule; hằng số → params; cấu trúc dữ liệu → schema.
  • Versioned rules: mọi RuleSet có effective_from / effective_to → trace được lương tháng 12/2025 tính bằng rule nào.
  • Deterministic & re-runnable: chạy lại cùng input luôn ra cùng output; hỗ trợ "what-if" khi audit.
  • Plug-in architecture: thêm công ty thứ 3 chỉ cần tạo newco.yaml + custom rule classes nếu có logic mới.

Vì sao schema này phục vụ cả 2?

  • WorkUnitType trung lập — dữ liệu mẫu: KG_MU_TAP, KG_MU_DONG, CONG_QUY_DOI, DAY_REGULAR, USD_BASE. Quasa dùng KG_MU_TAP + DAY_*; Mekong dùng CONG_QUY_DOI (sau Normalize) + KG_MU_DONG.
  • PayrollRun.run_type ∈ {MONTHLY, ADVANCE, SETTLEMENT, OFF_CYCLE}. Quasa chỉ dùng MONTHLY, Mekong dùng ADVANCE+SETTLEMENT.
  • Employee.classifier là string tự do → map về hạng A/B/C/D (Quasa) hay "cạo D2" (Mekong) qua rule.
  • MonthlyParams.params là JSON → linh động cho mọi hằng số theo kỳ (factor, DRC, FX, định mức công…).
  • Advance là entity chung: Quasa chưa dùng, Mekong dùng cho lương Đợt 1.
  • Employee.employee_type ∈ {DIRECT_WORKER, OFFICE_STAFF, INDIRECT}. Giai đoạn 1 chỉ xử lý DIRECT_WORKER. Các rule như PROGRESSIVE_TAX có applicable_to lọc theo field này, nên engine tự bỏ qua TNCN cho CN trực tiếp mà không cần sửa code.

// Cắt số thập phân (không làm tròn)
public static decimal TruncateDecimal(decimal value, int decimals)
{
    var factor = (decimal)Math.Pow(10, decimals);
    return Math.Truncate(value * factor) / factor;
}
